1. EXECUTIVE SUMMARY
======================================================================

RADAR will be a single-user, local-first web application designed to help a supervisor organize day-to-day work involving approximately 40 employees.

The application will focus on:

- A kanban-style task board as a primary working interface.
- A Today or Attention dashboard that identifies items needing action.
- A calendar view for task due dates, leave, and telework.
- Employee profiles that aggregate tasks, training, leave, telework, performance inputs, awards, and supervisory follow-ups.
- Fast capture of tasks, performance observations, and meeting notes.
- Recurring reminders for routine supervisory responsibilities.
- Local data storage with explicit export, backup, and recovery.
- No required server, cloud account, external API, or software installation on the operational NMCI computer.
- No runtime internet dependency.
- A production build that can be opened in an approved browser using local files, if NMCI browser policy permits.

The software will be developed on a personal or approved development computer using modern development tools. Only the compiled, self-contained production files will be moved to the NMCI environment through an authorized process.

The first development activity will not be the full application. It will be a small compatibility test that verifies what the NMCI browser allows when opening a local web application. This test will determine:

- Whether bundled JavaScript executes from a local file.
- Whether IndexedDB is available and persists between browser sessions.
- Whether download-based backup export works.
- Whether file import works.
- Whether drag-and-drop interactions work.
- Whether browser permissions or enterprise policies block required features.

The data architecture will be designed behind an abstraction layer so the storage implementation can be changed without rewriting the entire application.

2. PROBLEM STATEMENT
======================================================================

The supervisor currently manages work across multiple disconnected systems, including:

- Microsoft Planner for task tracking.
- Microsoft Excel for data tracking.
- ERP for timekeeping and leave.
- SWAT for command-specific workflows and information.
- Email, meetings, informal notes, and memory for follow-ups.
- Other systems that do not provide one consolidated supervisory view.

The authoritative data remains distributed across those systems. The supervisor repeatedly performs manual reconciliation and creates personal tracking mechanisms to remember:

- What needs to be done.
- Who is responsible.
- What is overdue.
- What is waiting on another person.
- Which employees need follow-up.
- Which training or telework items are approaching expiration.
- What employee accomplishments should be remembered during evaluations.
- Whether recurring supervisory actions have been completed.

The main problem is not the absence of source systems. The problem is the absence of a personal supervisory workspace that combines reminders, tasks, employee context, and structured notes in one place.

The application should therefore not attempt to replace ERP, SWAT, Planner, or official personnel systems. It should act as a supervisory cockpit that references and reconciles information from those systems.

======================================================================
3. PRODUCT VISION
======================================================================

Product vision statement:

RADAR is a private, local-first supervisory cockpit that helps one supervisor capture, organize, review, and complete work associated with employees, projects, requirements, meetings, and recurring responsibilities.

The application should make it easy to answer:

- What requires my attention today?
- What is overdue?
- What is due soon?
- What am I waiting on?
- Which employees have open actions?
- Which employee has not received recent supervisory attention?
- Which tasks have become stale?
- What training is approaching its due date?
- What telework agreements or requests require action?
- What leave may affect project staffing?
- What accomplishments should be captured for performance evaluations?
- What routine supervisory action should recur next?
- When did I last verify information against the authoritative system?

The long-term measure of success is simple:

The application becomes the first page opened at the beginning of the workday and the final page reviewed before ending the workday.

4. PRODUCT PRINCIPLES
======================================================================

4.1 Local first

The application must remain useful without network access. Data entry, search, filtering, task movement, reporting, and reminders must work locally.

4.2 No runtime dependencies

The operational build must not download scripts, fonts, icons, styles, telemetry, or other resources from the internet.

4.3 Fast capture

The user must be able to create a basic task in less than 15 seconds.

Minimum quick task flow:

1. Press New Task or a keyboard shortcut.
2. Enter a title.
3. Optionally select an employee.
4. Optionally select a due date.
5. Save.

4.4 Progressive detail

A task can begin with only a title. More information can be added later.

4.5 One underlying work model

Training actions, telework follow-ups, performance inputs, awards, and normal project tasks should share common work-item behavior where practical.

4.6 Source awareness

Records derived from another system should identify:

- Source system.
- Source reference.
- Date last verified.
- Verification status.

4.7 Action over documentation

The application should emphasize what needs attention rather than becoming a passive archive.

4.8 Explainable reminders

Every item in the Attention view should clearly state why it appears.

Examples:

- Overdue by 4 days.
- Due within 7 days.
- Waiting for 18 days.
- No activity for 30 days.
- Training expires in 21 days.
- No performance input recorded in 90 days.

4.9 Data portability

The user must be able to export all data in a documented, human-readable format.

4.10 Minimal sensitive data

Store only information necessary for legitimate supervisory organization.

4.11 Recoverability

A mistake, browser failure, corrupted database, or machine replacement must not permanently destroy the data if backups have been performed.

4.12 Desktop first

The primary target is a Windows desktop browser. Mobile support is not required for the initial version.

6. OPERATIONAL AND SECURITY BOUNDARIES
======================================================================

6.1 Development environment

Development should occur on a machine where the developer can use:

- Visual Studio Code.
- Git.
- Node.js.
- npm or another approved package manager.
- Codex, Claude Code, or another coding agent.
- Automated tests.
- Browser developer tools.

6.2 Operational environment

The target operational environment is an NMCI government computer with a managed browser, likely Microsoft Edge.

Assume:

- No administrator rights.
- No application installation.
- No local server.
- No command-line runtime.
- No internet dependency.
- Enterprise browser policies may disable storage or file features.
- The application may be opened through a file path.
- The browser may clear site data under some circumstances.
- Download locations and removable media may be restricted.
- Moving files to the NMCI computer must follow an approved process.

6.3 Information minimization

Do not store:

- Social Security numbers.
- DoD ID numbers unless specifically authorized and necessary.
- Home addresses.
- Personal phone numbers.
- Medical details.
- Family details.
- Leave medical explanations.
- Security clearance details.
- Passwords.
- Account credentials.
- Classified information.
- Sensitive operational details unrelated to supervisory tracking.
- Copies of official documents unless specifically authorized.
- Full personnel files.

Potentially reasonable fields include:

- Employee display name.
- Competency.
- Work role.
- Work email if permitted.
- Projects.
- Task assignments.
- Due dates.
- Broad training status.
- Broad leave dates required for workload awareness.
- Telework action status.
- Objective performance observations.
- Accomplishments.
- Award nomination status.
- Source-system reference information.
- Verification dates.

6.4 Neutral and objective notes

All notes should be:

- Work related.
- Objective.
- Dated.
- Concise.
- Attributable to a legitimate supervisory purpose.
- Written with the assumption that they may eventually be reviewed.

The application should not encourage unstructured personal judgments.

6.5 Data classification notice

Display a persistent footer or About notice:

“Use only for information authorized for local supervisory tracking. Do not enter classified information, credentials, medical details, Social Security numbers, or unnecessary personal information.”

6.6 No telemetry

The application must not include:

- Analytics.
- Crash reporting to an external server.
- Usage tracking.
- Remote logging.
- CDN requests.
- External fonts.
- External icons.
- Update checking.
- Advertising.
- Third-party scripts.

6.7 Content Security Policy

Where compatible with local execution, include a restrictive Content Security Policy that blocks external connections.

Target policy intent:

- Default deny external resources.
- Allow only bundled local scripts and styles.
- Disallow network connections.
- Disallow framing.
- Disallow plugins and object embedding.

Because local-file browser behavior varies, this policy must be tested during the compatibility spike.

7. CORE USER WORKFLOWS
======================================================================

7.1 Morning startup

1. Open the application.
2. See the Today or Attention dashboard.
3. Review overdue items.
4. Review items due soon.
5. Review stale waiting items.
6. Review upcoming leave and training.
7. Select the highest-priority work.
8. Move selected work to In Progress.

7.2 Quick task capture

1. Press the New Task button or keyboard shortcut.
2. Enter a title.
3. Select category.
4. Select employee, project, or both if relevant.
5. Select due date if known.
6. Save.
7. The card appears in Inbox by default.

7.3 Process Inbox

1. Open the Inbox column or Inbox view.
2. For each card:
   - Assign category.
   - Assign employee or project.
   - Set priority.
   - Set due date.
   - Move to Planned, In Progress, Waiting, or another status.
3. Leave unresolved cards in Inbox.

7.4 Update a task

1. Open a card.
2. Add note, checklist item, due date, or source reference.
3. Change status.
4. Save.
5. The system adds an activity entry.

7.5 Complete a task

1. Move the card to Complete or press Complete.
2. Record completion date automatically.
3. Optionally add a completion note.
4. If associated with an employee, ask:
   “Create a performance input from this completed work?”
5. If recurring, calculate or generate the next occurrence.
6. Keep completed card visible for a configurable period before archive.

7.6 Add a performance observation

1. Select Quick Add Performance Input.
2. Select employee.
3. Select date.
4. Select performance category or element.
5. Enter action or accomplishment.
6. Enter result.
7. Enter impact.
8. Link to project or completed task when applicable.
9. Save.

7.7 Employee review

1. Open Employee Directory.
2. Select an employee.
3. Review:
   - Open work.
   - Overdue work.
   - Recent accomplishments.
   - Upcoming leave.
   - Training.
   - Telework.
   - Awards.
   - Recent interactions.
4. Add follow-up task or performance input.

7.8 Training reconciliation

1. Open Training view.
2. Filter by due soon, overdue, or unverified.
3. Compare against SWAT or another authoritative system.
4. Update completion or verification date.
5. Add a follow-up task if information is missing.

7.9 Leave awareness

1. Open leave calendar or list.
2. Review planned absences.
3. Confirm project conflicts.
4. Add supervisory action tasks if needed.
5. Mark the record as verified against ERP.

7.10 End-of-day review

1. Review In Progress.
2. Move blocked work to Waiting.
3. Add next actions.
4. Complete finished cards.
5. Review tomorrow’s due items.
6. Export a backup if the backup reminder is active.

7.11 Weekly review

1. Review all Inbox items.
2. Review Waiting items older than a threshold.
3. Review stale tasks.
4. Review employees with no recent contact or performance input.
5. Review training due in the next 30 or 60 days.
6. Review upcoming leave and telework.
7. Review backup status.
8. Archive completed items older than the configured retention period.

8. RECOMMENDED TECHNICAL ARCHITECTURE
======================================================================

8.1 Architecture summary

Recommended development stack:

- Svelte for user interface components.
- TypeScript for type safety.
- Vite for development and production builds.
- IndexedDB for structured local persistence when supported.
- A lightweight IndexedDB wrapper such as Dexie, bundled into the build.
- Vitest or equivalent for unit tests.
- Playwright or equivalent for browser-level tests on the development computer.
- CSS variables and plain CSS for styling.
- Bundled SVG icons or locally defined icon components.
- No runtime CDN dependencies.

8.2 Why Svelte

Svelte is suitable because:

- The application is highly interactive.
- Components can remain compact.
- TypeScript support is strong.
- The production runtime can be small.
- It works well with forms, dialogs, dashboards, and state-driven interfaces.
- It can be compiled into static browser files.

React is also acceptable, but Svelte is the preferred starting recommendation.

Plain TypeScript is possible, but the application has enough screens, dialogs, forms, filters, and state transitions that a component framework will likely reduce maintenance effort.

8.3 Production artifact

Preferred production output:

supervisor-assistant/
    index.html
    assets/
        app.js
        app.css
        icons.svg
    README-RUN.txt

Optional production output:

supervisor-assistant.html

The single-file build is desirable for portability, but it must not become a hard requirement if it introduces unreliable storage or browser behavior. A small self-contained folder is acceptable.

8.4 No web server assumption

The production build should be designed to open directly from a local path.

Avoid features that require:

- Server-side routes.
- Server-side rendering.
- API endpoints.
- Service workers.
- WebAssembly.
- Cross-origin requests.
- Dynamic module loading.
- Absolute asset paths.
- Runtime environment variables.

8.5 Relative build paths

All production assets must use relative paths.

8.6 Storage abstraction

The rest of the application must not call IndexedDB directly.

Create a storage interface such as:

interface DataStore {
    initialize(): Promise<void>;
    getAllEmployees(): Promise<Employee[]>;
    saveEmployee(employee: Employee): Promise<void>;
    deleteEmployee(id: string): Promise<void>;
    getAllTasks(): Promise<Task[]>;
    saveTask(task: Task): Promise<void>;
    deleteTask(id: string): Promise<void>;
    exportDatabase(): Promise<BackupPackage>;
    importDatabase(data: BackupPackage): Promise<ImportResult>;
}

Possible implementations:

- IndexedDbDataStore
- InMemoryDataStore
- JsonFileDataStore, if a dependable file workflow becomes possible
- MockDataStore for testing

This separation is critical because NMCI policy may require a different storage strategy than the development computer.

8.7 Application state

Use a predictable state architecture:

- Persistent entities live in the DataStore.
- Derived views are computed from entities.
- UI-only state remains in component state or dedicated stores.
- Avoid duplicating persistent records in multiple state locations.
- All mutations pass through a service layer.
- Mutations produce activity-history entries.

8.8 Date handling

Store calendar dates as ISO date strings:

YYYY-MM-DD

Store timestamps as UTC ISO timestamps:

YYYY-MM-DDTHH:mm:ss.sssZ

Display dates in the user’s local format.

Avoid treating date-only values as midnight timestamps when possible. Leave dates, due dates, and training dates are calendar dates, not moments in time.

8.9 Identifiers

Use locally generated UUID-style identifiers.

Do not use array positions as identifiers.

8.10 Browser support target

Primary browser target:

- Current enterprise-managed Microsoft Edge available on the NMCI computer.

Secondary development targets:

- Current Microsoft Edge.
- Current Google Chrome.

The application does not need to support Internet Explorer.

Development computer:

1. Build the environment test.
2. Open through a local file path.
3. Run all tests.
4. Close browser.
5. Reopen.
6. Confirm persisted test data.
7. Export diagnostic report.

NMCI computer:

1. Transfer using an authorized process.
2. Open index.html.
3. Run all tests.
4. Record pass or fail.
5. Close browser completely.
6. Reopen the page.
7. Confirm persistence.
8. Export the diagnostic report if allowed.

9.4 Required result

Do not begin full persistence implementation until the following are known:

- Can the app execute?
- Can it load bundled assets?
- Can it persist structured data?
- Does persistence survive browser restart?
- Can data be exported?
- Can data be imported?
- Are local downloads permitted?
- Does moving the application folder affect stored data?
- Does changing the file path create a separate storage origin?
- Does the browser clear local data automatically?

9.5 Decision outcomes

Outcome A:
IndexedDB works reliably.

Proceed with IndexedDB as the primary working database and JSON as the backup format.

Outcome B:
IndexedDB works but persistence is unreliable.

Use IndexedDB only as a working cache. Require open/import at startup and export/save at shutdown.

Outcome C:
IndexedDB is blocked.

Use an explicit document workflow where the user opens a JSON data file, works in memory, and downloads a replacement JSON file when saving.

Outcome D:
JavaScript or local files are blocked.

The planned architecture is not viable in that environment. Reassess approved alternatives such as a static page hosted in an authorized location or a Navy-provided platform.

10. DATA STORAGE STRATEGY
======================================================================

10.1 Preferred strategy

Primary working storage:

- IndexedDB.

Canonical portable backup:

- JSON backup package.

10.2 Why IndexedDB

The data is structured and relational enough to benefit from:

- Indexed searches.
- Transactions.
- More capacity than basic key-value storage.
- Asynchronous operation.
- Offline availability.

10.3 Why JSON backup remains mandatory

Browser storage can be:

- Cleared.
- Disabled.
- Partitioned by origin.
- Affected by enterprise policy.
- Lost during browser profile replacement.
- Lost during machine replacement.
- Separated if the application file is moved.

Therefore, the application must treat exported backups as essential.

10.4 Backup package format

Suggested file name:

SupervisorAssistant_Backup_2026-07-04_0830.json

Top-level structure:

{
  "format": "SupervisorAssistantBackup",
  "formatVersion": 1,
  "applicationVersion": "0.1.0",
  "exportedAt": "2026-07-04T12:30:00.000Z",
  "databaseId": "uuid",
  "data": {
    "employees": [],
    "competencies": [],
    "projects": [],
    "tasks": [],
    "taskNotes": [],
    "taskChecklistItems": [],
    "performanceInputs": [],
    "trainingRequirements": [],
    "employeeTrainingRecords": [],
    "leaveRecords": [],
    "teleworkRecords": [],
    "awardRecords": [],
    "recurrenceTemplates": [],
    "savedViews": [],
    "activityEntries": [],
    "settings": {}
  },
  "integrity": {
    "recordCounts": {},
    "checksum": null
  }
}

10.5 Save-state indicator

The header should show:

- Data status: Saved.
- Data status: Saving.
- Data status: Save failed.
- Last local save time.
- Last external backup time.
- Backup recommended indicator.

10.6 Backup reminder policy

Default recommendation:

- Remind after 7 days without a backup.
- Remind after 50 changes without a backup.
- Remind before a database migration.
- Remind after a large import.
- Remind on application version change.

10.7 Import behavior

Import must never silently overwrite data.

Import workflow:

1. User selects backup.
2. Application validates format.
3. Application displays:
   - Backup date.
   - Application version.
   - Record counts.
   - Database ID.
4. User selects:
   - Replace current database.
   - Merge into current database.
   - Cancel.
5. Before Replace:
   - Offer to export current backup.
6. Perform transaction.
7. Display import report.
8. Record import activity.

10.8 Merge rules

Merge must use IDs.

When IDs conflict:

- If records are identical, skip.
- If incoming record is newer, show conflict.
- If current record is newer, show conflict.
- Do not silently choose unless a documented rule is selected.
- Allow export of conflict report.

10.9 Schema migrations

Every database version change must include:

- Migration number.
- Migration function.
- Pre-migration backup recommendation.
- Automated migration tests.
- Failure rollback where possible.

13. KANBAN BOARD SPECIFICATION
======================================================================

13.1 Default statuses

INBOX

Definition:
Captured but not fully processed.

Entry rules:
Any new task without an explicitly selected status enters Inbox.

PLANNED

Definition:
Defined work that is not currently active.

IN PROGRESS

Definition:
Work currently being performed.

WAITING

Definition:
Work blocked by another person, event, approval, system, or dependency.

Required field:
Waiting reason should be encouraged.

Optional fields:
- Waiting on.
- Follow-up date.

NEEDS REVIEW

Definition:
Work believed complete or ready but requiring review, verification, or approval.

COMPLETE

Definition:
Work has been completed.

Completion behavior:
- Set completion date.
- Offer performance input creation.
- Generate next recurrence if applicable.
- Keep visible for configured number of days.

13.2 Optional statuses

Later configuration may allow:

- Deferred.
- Cancelled.
- On Hold.
- Delegated.

Avoid fully customizable workflow in the first release. Too much configurability will complicate board logic.

13.3 Column limits

Optional work-in-progress limits:

- In Progress limit.
- Needs Review limit.

Initial implementation may only display warnings rather than block movement.

13.4 Drag-and-drop behavior

Requirements:

- Desktop mouse support.
- Visual drop target.
- Card placeholder.
- Cancel movement with Escape.
- Persist only after a valid drop.
- Activity entry records status change.
- Maintain keyboard-accessible movement alternative.
- Avoid losing edits during drag.

13.5 Card ordering

Each card has:

- status.
- boardOrder.

Use fractional or gap-based ordering to reduce mass updates.

Example:

1000, 2000, 3000.

When inserting between 1000 and 2000, use 1500.

Renormalize a column when gaps become too small.

13.6 Swimlanes

Optional board grouping:

- None.
- Employee.
- Competency.
- Project.
- Priority.
- Category.

Initial release can support only None and Employee if complexity is high.

13.7 Card color

Do not rely solely on color.

Use:

- Text labels.
- Icons.
- Borders.
- Badges.
- Accessible contrast.

Potential card indicators:

- Red overdue badge.
- Amber due-soon badge.
- Waiting clock icon.
- Training category label.
- Employee initials.

Colors should remain theme variables rather than hard-coded throughout components.

13.8 Completed-card behavior

Default:

- Remain in Complete column for 7 days.
- Then disappear from active board.
- Remain in archive and reports.

13.9 Bulk actions

Later MVP phase:

- Select multiple cards.
- Change status.
- Assign project.
- Assign employee.
- Set due date.
- Add tag.
- Archive.

Bulk delete should be avoided or require strong confirmation.

22. REMINDER AND ATTENTION ENGINE
======================================================================

22.1 Important limitation

A local browser application cannot be assumed to execute while closed.

The primary reminder mechanism is therefore:

- Calculate attention items when the application opens.
- Recalculate periodically while open.
- Display visible counters and banners.
- Allow optional calendar export later.

22.2 Attention rule model

Each attention item should contain:

- entityType.
- entityId.
- reasonCode.
- reasonText.
- severity.
- effectiveDate.
- sortScore.
- suggestedAction.

22.3 Default task rules

OVERDUE

Condition:
Task is incomplete and dueDate is before today.

DUE TODAY

Condition:
Task is incomplete and dueDate equals today.

DUE SOON

Condition:
Task is incomplete and dueDate is within configured threshold.

REMINDER REACHED

Condition:
Task is incomplete and reminderDate is today or earlier.

WAITING TOO LONG

Condition:
Task status is Waiting and last meaningful activity exceeds threshold.

STALE TASK

Condition:
Task is incomplete and updatedAt exceeds threshold.

FOLLOW-UP DATE REACHED

Condition:
Task has followUpDate today or earlier.

22.4 Default personnel rules

NO RECENT PERFORMANCE INPUT

Condition:
Active employee has no performance input within threshold.

NO RECENT CHECK-IN

Condition:
Active employee has no interaction within threshold.

MULTIPLE OVERDUE ITEMS

Condition:
Employee has more than configurable number of overdue tasks.

22.5 Default training rules

- Training overdue.
- Training due within 30 days.
- Training due within 14 days.
- Training due within 7 days.
- Training unverified beyond threshold.

22.6 Default leave rules

- Leave begins within configured number of days.
- Leave overlaps major project milestone.
- Leave record is unverified.

22.7 Default telework rules

- Request pending supervisor action.
- Agreement expires within configured threshold.
- Agreement expired.
- Record unverified.

22.8 Backup rules

- No backup within 7 days.
- More than 50 changes since last backup.
- Import completed without subsequent backup.
- Version migration completed without subsequent backup.

22.9 Attention severity

- Information.
- Low.
- Medium.
- High.
- Critical.

Critical should be used sparingly.

22.10 Sort scoring

Possible score inputs:

- Severity.
- Days overdue.
- Due today.
- Priority.
- Waiting age.
- Reminder date.
- Employee impact.
- Manually pinned.

Use deterministic scoring and allow manual sorting.

22.11 Snooze

Attention items may be snoozed without changing the underlying due date.

Snooze fields:

- snoozedUntil.
- snoozeReason.

Snooze options:

- Later today.

23. RECURRING TASK ENGINE

23.1 Recurrence template fields

23.2 Initial recurrence patterns

Support only simple patterns first:

- Daily.
- Weekly.
- Every N weeks.
- Monthly on a day number.
- Monthly on first, second, third, fourth, or last weekday.
- Quarterly.
- Annually.
- Every pay period, only if manually defined dates are available.
- After completion by N days or months.

23.3 Generation strategy

When the app opens:

1. Check active recurrence templates.
2. Generate missing occurrences through today plus advanceCreateDays.
3. Prevent duplicates using a recurrence instance key.
4. Update nextGenerationDate.
5. Add activity entries.

23.4 Recurrence instance key

Example:

templateId + scheduledDate + employeeId

This prevents duplicate creation.

23.5 Per-employee recurrence

A recurrence template can apply to:

- No employee.
- One employee.
- Selected employees.
- All active employees.
- All active employees in a competency.

Use caution:
Generating one card per employee may create too much clutter. Support a single supervisory task with an internal checklist as an alternative.

23.6 Completion-based recurrence

Example:

“Conduct employee check-in every 60 days after completion.”

After completion, calculate the next scheduled date from completedDate.

======================================================================
28. DATA VALIDATION RULES
======================================================================

28.1 General

- Trim leading and trailing whitespace.
- Reject empty required strings.
- Limit field lengths.
- Reject invalid dates.
- Prevent impossible date ranges.
- Escape and render all user text safely.
- Never interpret user-entered text as HTML.

28.2 Suggested limits

- Task title: 200 characters.
- Task description: 10,000 characters.
- Note: 10,000 characters.
- Employee name: 200 characters.
- Project name: 200 characters.
- Tags: 50 characters each.
- Maximum tags per record: 20.
- Performance fields: 10,000 characters each.
- Source reference: 500 characters.

28.3 Date rules

- endDate must be on or after startDate.
- completedDate should not be before created date unless imported historical data is explicitly allowed.
- expirationDate should not be before completedDate.
- dueDate can be historical for imported data.
- Calendar-date comparisons must use local date logic.

28.4 Referential integrity

- Task employeeId must reference an existing employee or be undefined.
- Task projectId must reference an existing project or be undefined.
- Related records must not reference permanently deleted entities.
- Archive parent entities rather than orphan children.

28.5 Duplicate detection

Warn on:

- Employee with same normalized name.
- Project with same normalized name.
- Training requirement with same normalized name.
- Duplicate recurrence instance key.

Warnings should not always block creation.

2. Begin transaction.
3. Save entity.

6. Update UI.

Never show Saved unless the persistence call succeeded.

38. CODING AGENT WORKING RULES
======================================================================

These rules should be placed in AGENTS.md, CLAUDE.md, or the equivalent repository instruction file.

1. Read NIWC_Supervisor_Assistant_Project_Plan.txt before making architectural changes.

2. Do not add any runtime network dependency.

3. Do not add telemetry, analytics, remote logging, or update checks.

4. Do not load scripts, styles, fonts, icons, or images from a CDN.

5. Do not add a backend server.

6. Do not require Node.js in the production environment.

7. Keep production assets usable from relative paths.

8. Do not use real employee names or real government information in tests, fixtures, screenshots, or documentation.

9. All persistent access must go through the DataStore interface.

10. Do not directly call IndexedDB from UI components.

11. Use TypeScript strict mode.

12. Add tests for new domain rules.

13. Treat all imported data as untrusted.

14. Never render user text as raw HTML.

15. Keep dependencies minimal and pinned.

16. Explain any new dependency before adding it.

17. Preserve data through schema migrations.

18. Do not silently delete or overwrite records.

19. Include an Undo path or confirmation for destructive actions.

20. Maintain keyboard accessibility.

21. Do not make color the only status indicator.

22. Use date-only logic for date-only fields.

23. Maintain an activity entry for important mutations.

24. Run:
   - type checking,
   - unit tests,
   - build,
   - no-network verification,
   before considering a task complete.

25. Update CHANGELOG.md for user-visible changes.

26. Update this plan when an architectural decision changes or when shipped scope materially changes.

27. Prefer small, reviewable commits.

28. Do not rewrite working modules solely for style.

29. When uncertain about NMCI compatibility, implement feature detection and a fallback.

30. Backup and restore are first-class features, not optional utilities.

41. RISKS AND MITIGATIONS
======================================================================

RISK 1:
NMCI blocks local JavaScript or storage.

Mitigation:
Run Phase 0 before full development. Preserve storage abstraction. Be prepared to use explicit JSON open/save workflow.

RISK 2:
Browser data is cleared.

Mitigation:
Mandatory backup feature, visible backup age, change-count reminder, emergency export.

RISK 3:
Manual entry burden becomes too high.

Mitigation:
Fast Quick Add, progressive detail, recurring templates, task-to-performance conversion, bulk updates, import options later.

RISK 4:
The application becomes another disconnected system.

Mitigation:
Always store source-system references and verification dates. Focus on actions and reminders rather than duplicating full authoritative records.

RISK 5:
Scope grows too quickly.

Mitigation:
Build phases in order. Do not add awards, attachments, integrations, or AI until core board and backup are used successfully.

RISK 6:
Sensitive information is entered.

Mitigation:
Persistent notice, field minimization, no unrestricted diary fields, no attachments, objective note guidance, sanitized test data.

RISK 7:
Single-file bundling becomes fragile.

Mitigation:
Support a small static folder as the primary fallback. Portability is more important than forcing one HTML file.

RISK 8:
Drag-and-drop is inaccessible or unreliable.

Mitigation:
Provide buttons and keyboard actions for all status changes.

RISK 9:
Date calculations are incorrect.

Mitigation:
Use date-only utilities, extensive boundary tests, avoid midnight conversions.

RISK 10:
Coding agents introduce unnecessary dependencies or architecture changes.

Mitigation:
Repository rules, pinned dependencies, small commits, plan review, automated no-network check.

RISK 11:
Generated code works only in development server mode.

Mitigation:
Test production output through local file path throughout development, not only at release.

RISK 12:
Data migrations corrupt existing records.

Mitigation:
Versioned migrations, pre-migration backup prompt, migration tests, rollback where possible.

RISK 13:
The application is useful only when open.

Mitigation:
Make startup review valuable. Consider calendar export later. Do not depend on background notifications.

RISK 14:
Performance notes become subjective or inappropriate.

Mitigation:
Structured Context, Action, Result, Impact fields and objective guidance.

RISK 15:
The project is abandoned before becoming useful.

Mitigation:
Phase 2 must deliver a usable employee list and kanban board before building specialized modules.

45. GLOSSARY
======================================================================

Attention item:
A computed indication that a record may require action.

Authoritative source:
The official system where a record is maintained.

Backup package:
A complete portable export of application data.

Board order:
A numeric value controlling card order within a kanban column.

Capability detection:
Runtime checks that determine whether a browser feature is available.

Competency:
An organizational grouping such as 55140 or 55230.

DataStore:
The application interface that isolates persistence technology from business logic and UI.

Due state:
A computed status such as Future, Due Soon, Due Today, or Overdue.

Employee 360:
A profile page aggregating records associated with one employee.

IndexedDB:
A browser-provided structured local database.

Local first:
An architecture where the application works primarily on the local computer without a network service.

NMCI compatibility spike:
A small test application used to validate the operating environment before full implementation.

Performance input:
A structured record of employee action, result, and impact for possible use in performance review preparation.

Recurrence instance:
A task generated from a recurring template for one scheduled occurrence.

Saved view:
A named set of filters, sorting, grouping, and display preferences.

Source reference:
A non-sensitive identifier or note that helps locate the authoritative record.

Stale task:
An incomplete task with no update for a configured time.

Verification date:
The date a local record was compared with its authoritative source.

Waiting age:
The amount of time a task has remained in Waiting status.
